# POR Usage Analysis

## Objective

The objective of this document is to study and document the existing Power-On Reset (POR) implementation in the VSDCaravel RTL, specifically:

- How `dummy_por` is implemented
- Where it is instantiated
- What signals it generates
- Which design blocks actually depend on POR versus a generic reset

This analysis is used as a basis for safely removing POR logic and replacing it with a single explicit external reset pin.

---

## 1. Overview of `dummy_por`

The POR logic in the original design is implemented using a module named `dummy_por`.

- Location
```bash
rtl/dummy_por.v
```
### Module Definition 

- From the dummy_por module we have 3 output signals `porb_h`, `porb_l`, `por_l`.

```bash
```verilog
module dummy_por(
    output porb_h,
    output porb_l,
    output por_l
);
```
---
### Behavioral Description

- From the RTL implementation:
    - dummy_por is purely a behavioral model
    - It does not observe any internal digital state
    - It models an analog RC delay using Verilog initial blocks and #delay

**Key code:**
```bash
initial begin
    inode <= 1'b0;
end

initial begin
    #500 inode <= 1'b1;
end
```

**This indicates:**

- POR is generated by a fixed time delay
- No dependency on clocks
- No dependency on logic state
- No dependency on reset sequencing

**The comments explicitly state:**
```bash
// This is a behavioral model!
// Actual circuit is a resistor dumping current onto a capacitor
```
---
### 2. POR Signal Definitions

- The dummy_por module generates three signals:
```bash
| Signal | Polarity     | Intended Domain | Description            |
|--------|--------------|-----------------|------------------------|
| porb_h | Active-low   | 3.3V            | Main reset signal      |
| porb_l | Active-low   | 1.8V            | Derived reset          |
| por_l  | Active-high  | 1.8V            | Inverted reset         |
```

**Signal Relationships**

- From the code:
```bash
assign porb_l = porb_h;
assign por_l  = ~porb_l;
```
**This proves:**

- porb_l is not independently generated
- por_l is simply a logical inversion
- No additional sequencing or logic exists
---
### 3. POR Instantiation and Distribution

- Location:
```bash
rtl/caravel_core.v
```

**Usage Pattern**

- dummy_por is instantiated once and its outputs are distributed throughout the design as reset inputs.
- There is no logic that distinguishes POR from reset — the POR signals are simply treated as asynchronous resets.
---
### 4. Housekeeping Logic Dependency

- Location:
```bash
rtl/housekeeping.v
```

**Example Reset Usage**
```bash
always @(posedge wb_clk_i or negedge porb) begin
    if (!porb) begin
        ...
    end
end
```

**This demonstrates:**

- porb is used strictly as an asynchronous reset
- There is no power-detection logic
- There is no POR-specific behavior
- The same structure is repeated for SPI and GPIO control logic.
---

### 5. Reset Distribution Summary
```bash
| Block            | Reset Used | POR-specific behavior |
|------------------|------------|-----------------------|
| Housekeeping     | porb       | No                    |
| SPI              | porb       | No                    |
| GPIO             | porb_h     | No                    |
| Management core  | rstb_h     | No                    |
```

- All logic treats POR signals as generic asynchronous reset inputs.
---
## Conclusion

- Dummy_por does not encode any functional reset sequencing
- POR signals are simple reset aliases
- No block requires POR-specific timing
- All logic is compatible with a single external reset pin

**This validates that POR can be safely removed without functional impact.**
